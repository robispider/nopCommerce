PROJECT BLUEPRINT: Unified B2B + B2C Marketplace Platform on nopCommerce

Executive Summary
Build an enterprise-grade, highly scalable multi-sided commerce platform using nopCommerce as the foundational commerce engine. The system integrates B2B Wholesale, B2C Retail, Dropshipping, Preorders, Escrow, and Storefront-as-a-Service into a single unified ecosystem.
The Golden Rule
Zero modification of nopCommerce core source code.
All functionality will be implemented via:
Custom Plugins
Event Consumers (IEventConsumer)
Dependency Injection (overriding default services like IOrderProcessingService)
Custom Action Filters and Action Results
Custom Database Tables (with FKs to core tables)
Widget Zones and UI overriding
Custom ViewEngines and IThemeContext overrides
Goal: Ensure 100% upgradeability with future nopCommerce releases with minimal merge effort.

Business Objectives
Objective 1 (Wholesale): Allow suppliers/manufacturers/importers to publish wholesale products with MOQs and dropship rules.
Objective 2 (Retail/SaaS): Allow resellers to operate branded, fully customized storefronts (e.g., shoe-king.com).
Objective 3 (Privacy): Absolute separation of B2B and B2C. Consumers never see wholesale pricing, supplier identities, or marketplace mechanics.
Objective 4 (Ecosystem): Combine the models of Alibaba (B2B), Shopify (SaaS Storefronts), and AliExpress (Dropship/Preorder) inside one nopCommerce instance.

Core Business Actors
Consumer: Browses B2C storefronts, places orders, tracks shipments. Restricted from all B2B areas via ACL.
Supplier: Accesses B2B portal. Creates wholesale listings, manages global stock, enables dropshipping/preorders, fulfills orders.
Reseller: Accesses B2B portal to source products, and B2C admin to manage their branded storefront, margins, and custom inventory.
(Note: A single account can act as both Supplier and Reseller).
Platform Admin: Manages verifications, escrow disputes, payout approvals, and global settings.

Architectural Principles & NopCommerce "Gotchas"
Principle 1: Multi-Vendor Over Multi-Store: Standard nopCommerce Store entities consume significant memory (caching settings per store). To support 10,000+ resellers, we will run a Single nopCommerce Store. Storefront separation will be handled via a custom Routing Engine mapping subdomains/domains to the native nopCommerce Vendor entity.
Principle 2: DI Override for Core Flows: We will override core services (e.g., Cart Validation, Order Processing) via Dependency Injection to handle marketplace-specific logic.
Principle 3: Horizontally Scalable: The architecture assumes load-balanced app servers, distributed caching, external object storage, and message queues from Day 1.

High-Level System Modules (Plugins)
Marketplace.Core: Shared interfaces, domain entities, standard events, and utilities. Dependency for all other plugins.
Marketplace.Business: Vendor onboarding, KYC/Verification workflows, document uploads, and ACL role assignment.
Marketplace.Storefront: Custom URL routing (IRouteProvider), dynamic theme context overrides, and domain mapping.
Marketplace.Wholesale: B2B catalog access, MOQ logic, bulk pricing tables, and supplier visibility rules.
Marketplace.Reseller: Product sourcing logic, margin/markup calculators, and Reseller dashboard.
Marketplace.Dropship: Supplier notification system, fulfillment tracking, and API endpoints for supplier dispatch.
Marketplace.Preorder: Lead-time calculators, preorder inventory buckets, and reservation statuses.
Marketplace.Escrow: Order state machines, funds holding logic, refund/dispute workflows.
Marketplace.Commission: Multi-tier fee calculators (fixed, percentage, category-specific) for revenue splitting.
Marketplace.Wallet: Internal ledger for Platform, Supplier, and Reseller balances, and withdrawal requests.
Marketplace.OrderSplitter: Logic to split mixed-carts into separate nopCommerce Order entities.
Marketplace.ApiIntegration: REST/GraphQL APIs for large B2B suppliers to sync inventory and pull dropship orders automatically.

Storefront & Routing Architecture
To bypass standard memory limits while allowing custom branding:
Routing Engine: Override IRouteProvider.
Read incoming request: shoeking.com or shoeking.platform.com or platform.com/store/shoeking.
Map URL to VendorId.
Inject VendorId into the current request context.
Theme Engine: Override IThemeContext.
Instead of separate physical theme folders, use one "Master Marketplace Theme".
Load Vendor-specific CSS variables (Logo, Primary Color, Font) from plugin tables and inject them into the header via Widget Zones based on the active VendorId.

Product Ownership Model ("Light Cloning")
Challenge: Heavy custom mapping tables for products will destroy nopCommerce search, cart, and price calculation performance.
Solution: The "Light Cloning" Model.
Supplier Product: Standard nopCommerce Product owned by Supplier Vendor. Invisible to B2C.
Reseller Product: Standard nopCommerce Product owned by Reseller Vendor. Fully indexed by nopCommerce for fast search/checkout.
The Link: A custom plugin table DropshipMapping (ResellerProductId -> SupplierProductId).
Data Sync: When a Supplier updates their Base Cost or Stock, a RabbitMQ event fires to update the cloned Reseller Products based on the Reseller's configured margin rules.
Benefit: B2C consumers experience native-speed search and checkout because the system queries native Product tables.

Order Flows & Cart Splitting
Because a consumer might add a dropshipped item (Supplier A), a local item (Reseller stock), and a Preorder item to one cart, we must split orders to maintain clean Escrow and Fulfillment data.
The Workflow:
Consumer checks out.
Override IOrderProcessingService.PlaceOrderAsync.
Analyze cart items via the DropshipMapping and Preorder tables.
Auto-Split: Generate multiple native nopCommerce Order records and group them via a custom MarketplaceOrderGroup tracking ID.
Order 1 (Dropship): Sent to Supplier A for fulfillment.
Order 2 (Local): Sent to Reseller for self-fulfillment.
Order 3 (Preorder): Placed into Escrow Holding State.

Escrow System & State Machine
Protects B2C consumers and ensures Suppliers only get paid upon successful delivery.
Escrow States:
Pending: Payment authorized, order placed.
Funded: Payment captured into Platform Wallet. Order sent to Supplier/Reseller.
Shipped: Tracking number uploaded.
Delivered: Courier API confirms delivery.
Released: Post-delivery grace period ends (e.g., 3 days). Funds moved to Supplier/Reseller Wallets.
Disputed: Admin intervention required.

16. Domain-Driven Design (DDD) Specification
Part 1: Bounded Contexts
The system is divided into logical boundaries. Each context encapsulates its own rules, data, and ubiquitous language, interacting with others strictly via interfaces or Domain Events.
1. Catalog Context
Responsibility: Manages B2C product visibility, search indexing (OpenSearch), and the "Light Clone" mapping mechanism.
Core Entities: Core Product, ProductCategory, ProductAttribute.
Interactions: Listens to Wholesale Context for B2B catalog updates; publishes indexing events.
2. Wholesale Context
Responsibility: Manages the B2B marketplace ecosystem, restricting access via ACL, enforcing Minimum Order Quantities (MOQ), bulk pricing tiers, and supplier catalog visibility.
Core Entities: Supplier Product Configurations, Tiered B2B Pricing.
Interactions: Controls what the Reseller Context is allowed to "see" and import.
3. Storefront Context
Responsibility: Manages multi-tenant routing without core multi-store bloat. Handles domain mapping, theme resolution, and B2C branding.
Core Entities: Domain Mapping Rules, Theme Configuration Variables.
Interactions: Intercepts HTTP requests and forces the Catalog Context to filter by the active Reseller.
4. Dropship Context
Responsibility: Links Reseller sales to Supplier fulfillment capabilities. Handles order routing and dispatch coordination.
Core Entities: Fulfillment Requests, Dropship Mappings.
Interactions: Reacts to Order Creation, interacts with Fulfillment Context.
5. Escrow Context
Responsibility: Financial safety mechanism. Runs the State Machine that locks B2C consumer funds until conditions (like delivery) are met.
Core Entities: Escrow Transaction, State Logs.
Interactions: Listens to Fulfillment events; triggers Commission/Wallet events upon release.
6. Wallet Context
Responsibility: Internal ledger system tracking virtual balances, credits, debits, and lock states.
Core Entities: Ledgers, Ledger Entries, Withdrawal Requests.
Interactions: The ultimate source of truth for vendor balances.
7. Commission Context
Responsibility: The math engine. Calculates platform fees, supplier cuts, and reseller margins per transaction.
Core Entities: Fee Schedules, Commission Rules.
Interactions: Queried by the Escrow Context prior to releasing funds to Wallets.
8. Fulfillment Context
Responsibility: Tracking physical movement. Interacts with courier APIs and manual tracking inputs.
Core Entities: Shipments, Delivery Proofs.
Interactions: Triggers state changes in Escrow Context upon confirmed delivery.
Part 2: Aggregate Roots
In the context of our nopCommerce plugins, an Aggregate Root often wraps a core ID (VendorId, ProductId) along with custom EF Core tables to enforce transactional consistency.
MarketplaceBusiness
Bounded Context: Business / Core
Description: The ultimate legal entity using the platform.
State / Properties: BusinessId (PK), VendorId (Core FK), TaxId, VerificationStatus (Pending/Approved/Rejected), RoleType (Supplier, Reseller, Both).
Behavior: ApproveKYC(), UpgradeToSupplier(), SuspendAccount().
Storefront
Bounded Context: Storefront
Description: The branded frontend configuration for a Reseller.
State / Properties: StorefrontId (PK), MarketplaceBusinessId, CustomDomain (e.g., shoeking.com), IsActive, ThemeVariables (JSON/Dictionary of colors, logos).
Behavior: MapDomain(), UpdateBranding(), GoOffline().
SupplierProduct
Bounded Context: Wholesale
Description: The master product offered to B2B buyers.
State / Properties: SupplierProductId (PK), CoreProductId (Core FK), SupplierBusinessId, BaseWholesalePrice, MinimumOrderQuantity, IsDropshipEnabled, IsPreorderEnabled, LeadTimeDays.
Behavior: UpdateWholesaleCost(), ToggleDropship(), AdjustLeadTime().
ResellerProduct
Bounded Context: Catalog / Dropship
Description: The "Light Clone" linking a Reseller to a Supplier's inventory.
State / Properties: ResellerProductId (PK), CoreProductId (Core FK - the customized clone), SupplierProductId (The source product), ResellerBusinessId, RetailMarginPolicy, SyncInventory (Boolean).
Behavior: CalculateRetailPrice(), SyncStockFromSupplier(), DisconnectFromSupplier().
MarketplaceOrderGroup
Bounded Context: OrderSplitting
Description: Represents a single B2C checkout session that was split into multiple core orders.
State / Properties: OrderGroupId (PK), CheckoutTransactionId, ConsumerCustomerId (Core FK), TotalAmountPaid, ChildOrderIds (List of Core FKs).
Behavior: AddSplitOrder(), MarkFullyProcessed().
EscrowTransaction
Bounded Context: Escrow
Description: The financial hold tied to a specific split order.
State / Properties: EscrowId (PK), CoreOrderId (Core FK), TotalLockedAmount, CurrentState (Pending, Funded, Shipped, Delivered, Released, Disputed), ReleaseDate.
Behavior: Fund(), MarkShipped(), InitiateDispute(), ReleaseFunds().
WalletAccount
Bounded Context: Wallet
Description: The financial ledger for a specific Marketplace Business.
State / Properties: WalletId (PK), MarketplaceBusinessId, AvailableBalance, LockedBalance (Pending withdrawals), CurrencyCode.
Behavior: Credit(), Debit(), LockFundsForWithdrawal(), ResolveWithdrawal().
Note: Operations on WalletAccount must be strictly ACID compliant and idempotent.
Part 3: Domain Events
Events trigger cross-context communication using nopCommerce's IEventPublisher. They ensure the system remains decoupled.
SupplierStockChanged
Triggered By: Supplier updates inventory count on their core Product.
Payload: SupplierProductId, NewStockQuantity, Timestamp.
Subscribers:
Catalog Context: Updates stock for all linked ResellerProduct clones.
Search Worker: Queues OpenSearch re-indexing.
ProductCloned
Triggered By: Reseller clicks "Add to Store" on a B2B catalog item.
Payload: SupplierProductId, ResellerBusinessId, NewCoreProductId (The generated clone).
Subscribers:
Dropship Context: Creates the ResellerProduct Aggregate linking the two IDs.
Storefront Context: Makes the product visible in the Reseller's custom domain mapping.
ResellerPriceChanged
Triggered By: Reseller manually updates their retail price, or an automated margin rule triggers.
Payload: ResellerProductId, OldPrice, NewPrice.
Subscribers:
Search Worker: Syncs the new price to OpenSearch for the B2C storefront frontend.
MarketplaceOrderCreated
Triggered By: Checkout completion, immediately after the custom Cart Splitter generates the child orders.
Payload: MarketplaceOrderGroup Aggregate, List of CoreOrderIds.
Subscribers:
Dropship Context: Evaluates which child orders are Dropship vs Local stock.
Escrow Context: Creates EscrowTransaction aggregates for the child orders.
DropshipOrderCreated
Triggered By: Dropship Context identifying a specific child order as a dropship requirement.
Payload: CoreOrderId, SupplierBusinessId, ResellerBusinessId, Items.
Subscribers:
Fulfillment Context: Generates a pending fulfillment request in the Supplier's B2B portal.
Notification Service: Emails Supplier "New Dropship Order Received".
ShipmentDelivered
Triggered By: Courier API webhook tracking success, or manual Supplier input.
Payload: CoreOrderId, TrackingNumber, DeliveryTimestamp.
Subscribers:
Escrow Context: Progresses EscrowTransaction to Delivered state and starts the grace period countdown timer (e.g., 72 hours).
Notification Service: Emails Consumer "Your order has arrived".
EscrowReleased
Triggered By: Grace period timer expiration without consumer dispute.
Payload: EscrowId, CoreOrderId, TotalAmount.
Subscribers:
Commission Context: Calculates who gets what (Platform cut vs. Supplier cut vs. Reseller cut).
Wallet Context: (Chained from Commission) Credits the respective WalletAccount balances.
WithdrawalRequested
Triggered By: Vendor requests a payout from their dashboard.
Payload: WalletId, MarketplaceBusinessId, RequestedAmount, PayoutMethod.
Subscribers:
Wallet Context: Moves funds from AvailableBalance to LockedBalance.
Admin Context: Flags the request in the Platform Admin dashboard for final manual/API bank payout approval.
