🏗️ PHASE 1: THE COMMERCE UNBLOCKERS (Gaps 1, 2, & 4)
Focus: Getting the cart, order, and inventory to safely interact in a highly concurrent multi-vendor environment.
1. Inventory Context Architecture (Gap 1)
To answer your specific design questions for Marketplace.Inventory:
Who is authoritative? Supplier Inventory is the Master. Reseller inventory is derived (unless the reseller physically pre-purchased the stock).
Race Condition Resolution (Supplier=5, Reseller A sells 3, Reseller B sells 4):
Inventory buckets must use Optimistic Concurrency (ConcurrencyVersion column).
If A and B checkout simultaneously, B's transaction will throw a DbUpdateConcurrencyException. The system catches this, re-reads the stock (now 2), and rejects B's checkout with "Only 2 remaining".
Reservation Timing:
Add to Cart: NO reservation (too much stock hoarding).
Enter Checkout (Payment Page): Soft Reservation (15-minute TTL via StockReservation table). AvailableQty decreases, ReservedQty increases.
Payment Success: Firm Reservation. ReservedQty decremented, AvailableQty stays deducted.
Order Cancelled / TTL Expires: AvailableQty restored.
Inventory Sync (Event-Driven Push):
Supplier updates stock natively -> SupplierStockChangedEvent published.
InventorySyncConsumer listens -> Updates Supplier InventoryBucket.
If SyncInventory = true on mappings -> Automatically updates linked Reseller InventoryBuckets.
2. Marketplace Order Domain (Gap 2)
The native nopCommerce Order is a B2C container. We must build the MarketplaceOrderGroup aggregate to sit alongside it.
New Entities to Implement:

OrderFinancialSummary: Immutable snapshot of totals (Subtotal, Total Commission, Total Tax, Shipping, Gateway Fees).
OrderFulfillmentSummary: Track aggregate states (e.g., 2 of 3 allocations shipped = "Partially Shipped").

⚙️ PHASE 2: THE CORE ENGINES (Gaps 3, 8, & 9)
Focus: Moving from CRUD to complex business logic execution.
4. The Application Services Layer (Gap 3)
These are the "Engines". You will build these specific interfaces and their implementations:
IOrderAllocationService: Takes a native cart, reads ProcurementPolicies and InventoryBuckets, and generates the MarketplaceOrderGroup.
IEscrowFundingService: Listens to payment success, locks funds.
IWalletSettlementService: Contains the strict TransactionScope(Serializable) logic for the two-phase commit.
IInventoryReservationService: Handles the 15-minute TTL logic and concurrency checks.
5. Commission Rule Evaluator (Gap 9)
The Commission Engine requires a strict Priority Resolution Matrix.
Rule Evaluator Logic (Highest to Lowest Priority):
SKU Override: Specific product has a negotiated rate (e.g., High-margin widget = 2%).
Vendor Specific: Platform negotiated a custom rate for Vendor ID 123 (e.g., 4%).
Category Specific: Electronics = 3%, Clothing = 7%.
Tier Specific: Vendor sold >$10k this month (e.g., Volume discount = 4.5%).
Global Default: Fallback (e.g., 5%).
Implementation Plan: The ICommissionService.CalculateSplitAsync() will load all rules for the Vendor/Product, sort by the Priority enum above, take the first match, and generate an immutable CommissionSplitResult stored against the Order.
6. Escrow FSM Implementation Matrix (Gap 8)
Here is the strict transition matrix you must implement in your EscrowStateMachine:
Current State	Valid Trigger Event	Next State	Financial Impact
Created	OrderPaidEvent	Funded	None (Held in Escrow Liability GL)
Funded	TicketAcceptedEvent	Processing	None
Processing	TicketShippedEvent	Shipped	None
Shipped	ShipmentDeliveredEvent	Delivered	None (Starts 72h Grace Period)
Delivered	72h Timer Expires	SettlementPending	Triggers SettlementRequestedEvent
Delivered	DisputeRaisedEvent	Disputed	Pauses 72h Timer
SettlementPending	WalletSettledEvent	Settled	CRITICAL: Funds hit wallets
Disputed	DisputeResolved_Release	SettlementPending	Resumes settlement path
Disputed	DisputeResolved_Refund	Refunded	Platform initiates refund
