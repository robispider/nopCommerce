2. The Enhanced Escrow State Machine
Escrow purely controls when the Ledger is allowed to move money. It does not hold the money itself.
A. The Expanded States
CREATED (Cart split, waiting for payment confirmation)
FUNDED (Payment cleared, locked in system)
PROCESSING (Supplier acknowledges and is sourcing/packing)
SHIPPED (Supplier uploaded tracking)
DELIVERED (Courier webhook confirms delivery)
GRACE_PERIOD (3-day countdown begins for customer disputes)
RELEASED (Terminal state: Triggers Ledger Credits)
DISPUTED (Customer raises issue. Timer paused. Requires Admin resolution)
REFUNDED (Terminal state: Customer gets money back)
CANCELLED (Terminal state: Supplier didn't ship in time)
B. The Audit Trail
EscrowStateHistory
Never overwrite a state without logging it.
Id, EscrowTransactionId
OldStateId, NewStateId
ChangedBy (UserId or System)
ChangedAtUtc
Notes (e.g., "Courier Webhook confirmed via SSL")
C. Concurrency Protection (The Race Condition Fix)
To prevent a "Release Job" and "Customer Dispute" from executing at the exact same millisecond:
All state updates will use Optimistic Concurrency via a RowVersion (or DB-level transaction locks).
Logic: UPDATE Escrow SET State = 'DISPUTED' WHERE Id = 123 AND State = 'GRACE_PERIOD' AND RowVersion = @OldVersion
If AffectedRows == 0, the transaction aborts. No double execution.

Id, RuleName (e.g., "Default", "Electronics", "Gold Vendor")
CategoryId (Nullable)

Flow: When an Escrow reaches RELEASED, the Escrow Engine asks the ICommissionService -> "What is the split for this order?". The service evaluates the rules, calculates the numbers, and commands the WalletLedger to write the entries.
4. Withdrawal & Refund State Machines
Because API calls to banks and gateways fail frequently, we need state machines for money leaving the system.
A. Withdrawal Workflow (Vendor paying out)
When a Reseller asks for their ৳1,250:
REQUESTED: Ledger writes a Debit to AvailableBalance and Credit to PendingBalance. (Money is locked, preventing double withdrawal).
PROCESSING: Admin approves, system pings bank API.
COMPLETED: Bank confirms. Ledger clears the PendingBalance.
FAILED: Bank rejects. Ledger Debits Pending and Credits Available (money returned to vendor).
B. Refund Workflow (Customer getting money back)
When an Escrow is REFUNDED:
REQUESTED: Ping payment gateway (SSLCommerz/bKash).
PROCESSING: Waiting for gateway async confirmation.
COMPLETED: Gateway confirms. Escrow closed.
FAILED: Gateway error. Manual admin intervention required.
5. Technical Safeguards (The "Must Haves" for the AI Coder)
When we begin generating the C# code for this, the AI/Developer must follow these strict rules:
Idempotency Keys: Every LedgerEntry insertion method MUST accept an IdempotencyKey (e.g., EscrowRelease_Order123_Vendor456). Before inserting, query the ledger. If the key exists, return success immediately without inserting again.
Transaction Scopes: A state change in EscrowTransaction, writing to EscrowStateHistory, and updating the WalletLedger MUST happen inside a single EF Core IDbContextTransaction. If the server loses power midway, the whole thing rolls back.
Cron Job Fallbacks: Auto-delivery (moving SHIPPED to DELIVERED without a webhook) will ONLY trigger if: Tracking exists + Courier API is unresponsive + Supplier Lead Time is heavily exceeded.
